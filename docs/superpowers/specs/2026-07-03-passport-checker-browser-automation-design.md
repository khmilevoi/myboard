# Passport Checker and Browser Automation Runtime Design

**Date:** 2026-07-03
**Status:** Approved

## Goal

Add a personal passport-checker widget that submits the configured Ukrainian
passport booklet series and number to `https://pasport.org.ua/solutions/checker`
and displays the validated JSON status returned by the checker.

The integration must work in the production Docker Compose stack on a Raspberry
Pi that is administered through SSH only. It must survive Cloudflare session
expiry without requiring a browser extension, persist the browser session across
deployments, and establish reusable infrastructure for future browser-driven
widget tasks.

## Scope

This design includes two tightly coupled deliverables:

1. A reusable `browser-automation` workspace package and Docker service that
   executes allowlisted widget-owned browser tasks in a persistent Chromium
   profile.
2. A `passport-checker` widget whose first version checks one preconfigured
   Ukrainian passport booklet using the fixed checker fields `service=1` and
   `doc_1_select=1`.

The infrastructure and widget are delivered together because the widget is the
first production consumer and provides the acceptance path for the runtime.

## Non-goals

- No generic remote-browser API that accepts arbitrary URLs or JavaScript.
- No browser extension, userscript, hidden iframe, or copied browser cookies.
- No direct Node.js request to the protected checker endpoint.
- No support in the first version for ID cards, foreign passports, multiple
  saved documents, or arbitrary checker services.
- No persistence of passport data in Valkey, IndexedDB, widget storage, browser
  local storage, application logs, or RPC payloads.
- No automated Cloudflare challenge solving.
- No live calls to `pasport.org.ua` from automated tests or CI.

## Technical Findings

The hidden-iframe approach is not viable. The checker currently sits behind a
Cloudflare challenge and returns `X-Frame-Options: SAMEORIGIN`. Even if a
cross-origin HTML form submission were accepted, the parent page could not read
the resulting document because iframe access is restricted by the same-origin
policy. Cross-origin `postMessage` would require cooperation from code running
on the remote origin.

A direct client `fetch` is also not a reliable integration boundary. Browser
code cannot manually reproduce headers such as `Cookie`, `Referer`, or `Sec-*`;
credentialed cross-origin reads require explicit compatible CORS responses; and
`no-cors` produces an unreadable opaque response. The multipart boundary must be
generated by the browser from `FormData`.

The selected solution therefore executes a same-origin `fetch` inside a real
page loaded from `pasport.org.ua`. Playwright supports persistent contexts,
headed execution on Linux through Xvfb, ARM64 Linux, and Docker deployment. The
browser package version and Docker image version must remain identical.

References:

- [MDN same-origin policy](https://developer.mozilla.org/en-US/docs/Web/Security/Same-origin_policy)
- [MDN CORS guide](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/CORS)
- [Playwright persistent context](https://playwright.dev/docs/api/class-browsertype#browser-type-launch-persistent-context)
- [Playwright Docker guidance](https://playwright.dev/docs/docker)
- [Docker Compose secrets](https://docs.docker.com/compose/how-tos/use-secrets/)

## Architecture

```text
passport-checker UI
        |
        | existing typed widget RPC
        v
myboard server + widget server handler
        |
        | internal typed HTTP task request
        v
browser-automation service
        |
        | registered passport-checker/check task
        v
persistent headed Chromium under Xvfb
        |
        | same-origin FormData fetch
        v
pasport.org.ua/solutions/checker
```

The production stack gains one independent `browser-automation` service. It
owns Chromium, the persistent profile, task scheduling, task validation, and
the optional noVNC recovery surface. The existing server remains a lightweight
Node/Alpine service and never imports Playwright.

The server invokes automation through an internal client injected into
`WidgetServerContext`. The client is an adapter: widget server handlers see a
typed error-as-value API, not HTTP details. The automation service is reachable
only on the Compose network. It has no public route through the client nginx
service.

## Widget-owned Browser Entry Points

A widget may add an optional root `browser.ts` entry point beside its existing
`client.ts` and `server.ts` files. A browser entry point declares named task
schemas and handlers without importing client or React code.

Browser codegen scans widget directories and generates a readable production
task registry for the automation service. The widget directory basename remains
the canonical widget ID. Widgets without `browser.ts` produce no browser tasks.
The generated registry follows the existing server-registry separation: browser
builds load browser entry points only, and server/client builds do not load them.

The runtime dispatch key is `(widgetId, taskId)`. Both request and result are
validated with Zod before and after handler execution. Unknown widgets, unknown
tasks, invalid payloads, and invalid results are rejected before crossing the
service boundary.

The HTTP protocol never accepts an executable script, selector set, browser
launch argument, target origin, or arbitrary URL. Adding a new automation flow
requires adding reviewed source code and registering it through codegen.

## Server Integration

`WidgetServerContext.api` gains a browser capability with an invocation shaped
around the current widget ID:

```text
browser.invoke(taskId, payload, resultSchema) -> Error | validated result
```

The server implementation sends an internal request to the automation service,
applies a bounded timeout, validates the response envelope, and maps transport
failures to tagged `errore` errors. The widget handler calls only its own task;
the current widget type is taken from the server context rather than accepted
from client input.

`BROWSER_AUTOMATION_URL` configures the internal service URL. When it is absent
or unreachable, the board and other widgets continue to operate; only browser
tasks return `BrowserUnavailableError`. The main server must not depend on
browser readiness for its own healthcheck.

## Browser Runtime Lifecycle

The automation service launches one headed Chromium persistent context with a
dedicated user-data directory. Chromium runs with `headless: false` under Xvfb
even during unattended operation. This keeps the executable, rendering mode,
profile, and browser-visible fingerprint consistent between unattended requests
and manual Cloudflare recovery.

The browser profile lives in a named Compose volume. It is never the default
Chrome profile, is not copied from another machine, and is opened by only one
Chromium process. The volume survives image rebuilds and container restarts.

All tasks for the profile pass through one FIFO queue. The first version runs
one task at a time to avoid profile locking, page interference, excess Raspberry
Pi memory use, and ambiguous cancellation. Each task has a deadline. Timeout or
browser failure terminates the current task but does not trigger an automatic
checker retry. The caller may retry explicitly.

The runtime exposes separate process health and browser-session state:

- `/health` reports whether the service and dispatcher are alive. A Cloudflare
  challenge does not make this endpoint unhealthy and must not cause a restart
  loop.
- Task responses report whether manual session recovery is required.

The service closes pages and the persistent context during graceful shutdown.
Unexpected Chromium exits fail the active task, recreate the context from the
same persistent volume, and leave retry control to the caller.

## Passport Checker Task

The task ID is `passport-checker/check`. It accepts an empty object because the
document identity is provisioned as deployment secrets and is never supplied by
the client or main server.

The task performs these steps:

1. Read `passport-checker_series` and `passport-checker_number` through the
   runtime's scoped secret reader.
2. Validate the series as exactly two Ukrainian Cyrillic uppercase letters and
   the number as exactly six ASCII digits. Secret values are never included in
   validation messages.
3. Reuse or create a page in the persistent context and navigate to
   `https://pasport.org.ua/solutions/checker`.
4. Detect a Cloudflare challenge or blocked response before submission. Return
   `BrowserSessionRequiredError` rather than attempting to bypass it.
5. In the page context, construct `FormData` with `service=1`,
   `doc_1_select=1`, `doc_1_series`, and `doc_1_number6`.
6. Execute `fetch('/solutions/checker', { method: 'POST', body: formData })`.
   The browser supplies cookies, referrer metadata, user-agent metadata, and the
   multipart boundary.
7. Check the HTTP status, parse JSON, and validate a result containing an
   integer `status` and string `send_status_msg`.
8. Return only the validated checker result.

The task does not fill visible DOM inputs because the same-origin request is the
smaller and less selector-sensitive integration. Navigation still initializes
the origin, session, and Cloudflare state before the request.

## Secret Provisioning

The passport series and number are configured once in the deployment `.env`
used by the `pi` CLI:

```text
PASSPORT_SERIES=<two-letter series>
PASSPORT_NUMBER=<six-digit number>
AUTOMATION_SSH_TARGET=<existing SSH target for the Raspberry Pi>
```

`rpi.toml` declares `[env] file = ".env"`. The operator provisions or updates
the values with `pi env send`, using `pi env send --apply` when the running stack
must be restarted.

The Compose file declares runtime secrets whose source is the deployment
environment and grants them only to `browser-automation`. Compose mounts them as
files under `/run/secrets`; they are not included in the browser container's
environment. The task accesses them through scoped names and reads them only for
the duration of an invocation.

`AUTOMATION_SSH_TARGET` is non-secret operational configuration. Compose passes
it to `browser-automation`, which includes it only in the safe public metadata of
a `BrowserSessionRequiredError`; the passport secret files are never included in
that response.

`.env` remains ignored. The repository contains only non-secret example keys and
operator documentation. Passport values must not appear in image layers,
build arguments, Compose command lines, logs, test fixtures, screenshots, traces,
or error serialization.

## Cloudflare Recovery Through SSH

The browser container includes Xvfb plus an x11vnc/noVNC bridge. Compose publishes
noVNC only on the Raspberry Pi loopback interface:

```text
127.0.0.1:6080:6080
```

When a task returns `BrowserSessionRequiredError`, the widget displays a short
recovery sequence using the configured `AUTOMATION_SSH_TARGET`:

1. Open an SSH local-forward from the operator machine:
   `ssh -L 6080:127.0.0.1:6080 $AUTOMATION_SSH_TARGET`.
2. Open `http://127.0.0.1:6080` locally.
3. Complete the challenge in the already-running persistent Chromium session.
4. Close the tunnel and press `Retry` in the widget.

The noVNC port is not exposed on the LAN or public ingress. SSH authentication is
the access boundary. The same browser process and profile remain active during
manual recovery.

## Widget Model and UI

The widget has no passport input fields in normal operation. It renders:

- a title and concise description;
- a primary `Check` button;
- pending state while the task is active;
- the checker's validated `send_status_msg` and numeric status after success;
- a session-recovery panel for `browser_session_required`;
- safe retryable messages for timeout and temporary unavailability.

The model uses a named Reatom async action extended with `withAsync`. Request
state, derived display state, timeout/cancellation handling, and error mapping
remain in `model/`; React owns only DOM interaction and small view glue. Exported
React function components use `reatomMemo`.

The submit button remains available for native focus and validation semantics,
then becomes disabled while a valid request is pending to prevent duplicate
queue entries. There is no automatic polling or retry. The last result exists
only in in-memory Reatom state and disappears on page reload.

## Error Model

Expected failures are tagged errors returned as values. External throwing
boundaries are converted immediately with `.catch(...cause...)`, and callers use
flat `instanceof Error` early returns.

The public categories are:

- `BrowserUnavailableError`: the internal automation service cannot be reached.
- `BrowserSessionRequiredError`: Cloudflare or an unauthenticated browser state
  requires operator attention.
- `AutomationTimeoutError`: queue wait, navigation, or execution exceeded the
  configured deadline.
- `UpstreamResponseError`: the checker returned a non-success HTTP response.
- `InvalidCheckerResponseError`: the response was not valid expected JSON.
- `BrowserConfigurationError`: required document secrets are missing or invalid.

The automation service returns a stable machine code and safe public message.
The main server preserves the machine code through widget RPC. Raw Playwright,
network, Zod, and upstream response details stay in internal cause chains and
are logged only after redaction. Request and response bodies are not logged.

## Docker and Raspberry Pi Deployment

`browser-automation` uses a Dockerfile based on the exact Playwright Ubuntu
image corresponding to the workspace Playwright version. The service uses
`init: true` and Chromium-compatible shared-memory configuration. It runs as the
image's non-root browser user after image setup, and the profile volume is
writable only by that runtime user.

Compose adds:

- the internal browser API port exposed only to other services;
- the loopback-only noVNC port;
- a named persistent browser-profile volume;
- runtime secrets sourced from the deployment environment;
- a process healthcheck;
- `restart: unless-stopped`.

The server receives only `BROWSER_AUTOMATION_URL`; it receives neither passport
secret. The client and nginx images do not contain Playwright or browser task
code. The browser image does not import widget client or React modules.

The production build on Raspberry Pi generates the browser registry inside the
browser image, just as the server image generates its server registry. Build
timeouts retain the existing 30-minute Pi allowance unless measurement shows
that the additional image exceeds it.

## Development Workflow

The browser package supports a local development entry point with an ignored
development profile directory. Docker development adds the same service,
profile volume, and fake/non-production secret inputs. The main server may run
without the service; browser-backed widgets then render the unavailable state.

Codegen gains a browser target and includes it in workspace-wide `codegen`,
tests, and typechecks. Client codegen remains independent so frontend-only work
does not launch or import browser code.

## Testing Strategy

No automated test contacts the real checker.

### Contract and dispatcher tests

- generated browser definitions contain only widgets with `browser.ts`;
- duplicate widget/task IDs are rejected;
- unknown tasks and invalid payloads are rejected;
- handler results are validated;
- error envelopes contain stable codes and no sensitive values.

### Runtime tests

- FIFO execution permits only one active task per profile;
- queue wait and execution deadlines return typed timeouts;
- browser crash and shutdown clean up active work;
- a challenge state does not fail `/health`;
- logs and serialized errors redact task payloads and secret values.

### Passport task browser tests

A local fixture server provides a checker page and same-origin POST endpoint.
Tests cover:

- correct `FormData` field names and values sourced from injected fake secrets;
- browser-generated multipart requests;
- valid success JSON;
- non-success HTTP responses;
- invalid JSON and schema mismatch;
- a fixture challenge page producing `BrowserSessionRequiredError`.

### Server and widget tests

- the internal automation client maps connection, timeout, and response errors;
- widget RPC exposes safe codes without raw causes;
- the Reatom model handles pending, success, retry, and every public error;
- component tests cover the success and SSH-recovery views;
- no test snapshot or assertion output includes real document data.

### Verification gates

- targeted package tests and typechecks;
- workspace `pnpm test` and `pnpm typecheck`;
- client, server, widget, and browser image builds;
- `docker compose config` with non-production secret values;
- a manual ARM64 Raspberry Pi smoke test.

## Manual Acceptance on Raspberry Pi

The feature is accepted when all of the following are demonstrated:

1. `pi env send` provisions the document and a subsequent deploy starts the full
   stack without committing a secret file.
2. The browser profile survives a browser-image rebuild.
3. With an expired session, the widget reports that browser attention is
   required while the main board remains healthy.
4. The operator reaches noVNC only through the SSH tunnel and completes the
   challenge in the persistent browser.
5. Retrying the widget returns the checker's validated status and message.
6. Container inspection, normal logs, Valkey, and browser-visible widget state
   do not reveal the passport series or number.

## Risks and Mitigations

- **Cloudflare changes or automation detection:** do not attempt bypasses; expose
  explicit manual recovery and treat the upstream as inherently brittle.
- **Checker HTML changes:** the task uses same-origin fetch rather than DOM
  selectors; only the endpoint and response contract remain coupled.
- **Raspberry Pi resource pressure:** keep one browser context and sequential
  tasks, close unused pages, and isolate Chromium from the main server.
- **Profile corruption:** preserve the volume, shut down gracefully, and surface
  a configuration/recovery error rather than silently deleting the profile.
- **noVNC exposure:** bind to host loopback only and require SSH forwarding.
- **Secret leakage:** use Compose runtime secrets, empty task payloads, redacted
  logs, fake test secrets, and no persistence of results or identifiers.
- **Infrastructure coupling:** the main server remains healthy without browser
  automation; browser failures affect only widgets that explicitly invoke it.

## Decision Summary

The implementation will use a dedicated Playwright sidecar with a persistent
headed Chromium session under Xvfb, recoverable through SSH-tunneled noVNC. A
typed, allowlisted widget browser-task registry provides reusable infrastructure
without exposing a general browser-control API. The passport identity is
provisioned once as Compose runtime secrets through `pi env send`, and the widget
itself sends no passport data.
